# Contribuer à L'Anthropie

L'édifice est CC0 1.0 Universal. Personne n'en est propriétaire. Toute contribution est bienvenue selon les modes décrits ci-dessous.

## Cinq modes de contribution

### 1. Lire et signaler

Lecture critique des couches qui résonnent avec votre travail. Ouvrir une **Issue GitHub** pour :

- Signaler une référence scientifique manquante ou erronée
- Proposer un cadre alternatif non considéré
- Pointer une appropriation culturelle problématique
- Soulever une contradiction interne entre couches
- Demander précision sur un mécanisme

Les Issues sont publiques. La discussion est publique. Aucune autorité centrale ne tranche — la communauté discute.

### 2. Co-designer une couche

Si une couche aligne avec votre travail (académique, praticien, juridique, artistique), proposer modifications via **Pull Request** :

- Fork le repo
- Modifier le fichier de couche concernée (et seulement celui-là, sauf cas exceptionnel)
- Documenter dans le commit message la justification (sources, expérience, raisonnement)
- Ouvrir PR avec contexte clair

Pour les Pull Requests touchant aux couches mobilisant traditions vivantes (autochtones, religieuses, culturelles spécifiques), voir section "Co-design avec communautés" ci-dessous.

### 3. Prototyper localement

L'édifice gagne à être incarné, pas seulement lu. Si vous montez un prototype d'une couche dans votre contexte :

- Documenter ouvertement (blog, vidéo, papier)
- Référencer le repo
- Partager retours dans Issues ou Discussions GitHub
- Si vous voulez, ajouter votre prototype à une page `PROTOTYPES.md` (à créer)

Aucune certification, aucune licence requise. Vous prototypez sous votre responsabilité, dans votre contexte, avec vos termes.

### 4. Plaidoyer institutionnel

Si vous êtes en position d'introduire l'édifice dans votre réseau (université, fondation, ONG, gouvernement, conférence), faites-le. Aucune permission requise. Aucune attribution exigée. Si l'édifice est récupéré par d'autres porteurs sans mention de l'origine, c'est un succès.

## Co-design avec communautés (charte anti-appropriation)

L'édifice mobilise des références aux traditions vivantes (Maori, Anishinaabe, Aboriginal, Sami, Yoruba, Lakota, autres). Toute adaptation/modification d'une couche pour une tradition spécifique respecte ces principes (cf. Couche 7 Strate 5) :

1. **Initiative de la communauté.** Aucune adaptation "Maori-inspired", "vision quest-style" ou similaire ne peut être proposée par contributeur extérieur sans qu'une délégation reconnue de la communauté concernée ne l'initie.

2. **Rémunération obligatoire des co-designers.** Au tarif Compagnon-Maître senior + reconnaissance d'auteur dans le fichier modifié.

3. **Souveraineté communautaire.** La communauté peut retirer son consentement à tout moment. La version co-designée doit alors être retirée du repo sous 90 jours.

4. **Pas de transposition extra-communautaire.** Une version co-designée avec communauté Lakota n'est PAS proposable à un adolescent non-Lakota sans nouvelle autorisation.

Si vous représentez une communauté souhaitant co-designer une couche, ouvrir une Issue avec tag `co-design-souverain` ou écrire à `contact@anthropie.org`.

## Style et exigences

- **Sobriété.** Pas de marketing, pas de hype, pas d'auto-promotion. Le contenu suffit.
- **Sources rigoureuses.** Toute claim empirique référencée. Pas de "selon les recherches" sans citation précise.
- **Honnêteté assumée.** Sections "Angles morts résiduels" obligatoires. Si une modification résout un angle mort, elle doit ouvrir un autre.
- **Anti-purity.** L'édifice n'est pas pur. Les contradictions internes assumées sont normales. Une PR cherchant à "harmoniser" tout sera examinée avec scepticisme.
- **Anti-marque.** Aucun ajout de "Anthropie™", "Anthropie Method", ou marketing similaire. CC0 strict.

## Conduite

Voir [CODE_OF_CONDUCT](CODE_OF_CONDUCT.md). En résumé : critique radicale bienvenue, attaque personnelle interdite, respect souveraineté autochtone non-négociable.

## Gouvernance (provisoire)

Tant qu'aucune fédération n'est constituée formellement, les Pull Requests sont mergées par les mainteneurs initiaux du repo après discussion publique d'au moins 7 jours dans Issues. À terme, gouvernance prévue par tirage au sort + rotation parmi contributeurs actifs (modèle Sortition Foundation).

L'objectif est l'auto-effacement progressif des mainteneurs initiaux au profit de la communauté. Si l'édifice n'est plus attribuable à personne dans 5-10 ans, c'est le succès.
